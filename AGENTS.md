# AGENTS.md

คู่มือนี้ใช้เป็นกติกาการทำงานสำหรับ AI agent ที่เข้ามาช่วยพัฒนาโปรเจกต์ APS (Academic Planning Support)

## ภาพรวมโปรเจกต์

APS เป็นเว็บแอป Next.js สำหรับช่วยนักศึกษาวางแผนการเรียนตามหลักสูตร โดยใช้ข้อมูลผลการเรียน หลักสูตร รายวิชา วิชาบังคับก่อน แผนรายเทอม และวิชาที่เปิดสอน เพื่อวิเคราะห์สถานะการเรียน คาดการณ์วันจบ และจำลองผลกระทบจากการถอน/ลงเพิ่ม/ไม่ผ่านรายวิชา

ผู้ใช้หลักมี 2 กลุ่ม:

- นักศึกษา: ดู dashboard, จัดการข้อมูลผลการเรียน, ตรวจคาดการณ์วันจบ, ดูแผน 8 ปี, จำลองสถานการณ์
- ผู้ดูแลระบบ: จัดการหลักสูตร รายวิชา prerequisite แผนรายเทอม วิชาที่เปิดสอน และ import CSV

## กติกาก่อนแก้โค้ด

1. อย่าเริ่มเขียนโค้ดทันที
2. วิเคราะห์ requirement ให้ชัดก่อนเสมอ
3. ถ้า requirement กว้าง ให้เขียนเป็น bullet ที่ตรวจสอบได้
4. วางแผนก่อนว่าไฟล์ใดจะถูกสร้างหรือแก้ไข
5. อธิบายหน้าที่ของแต่ละไฟล์ก่อนลงมือแก้
6. ทำทีละ feature หรือทีละ step เล็ก ๆ
7. ห้ามลบโค้ดเดิมหรือ revert งานค้างโดยไม่จำเป็น
8. หากพบไฟล์ที่มีการแก้ค้างอยู่ ให้ถือว่าเป็นงานของผู้ใช้หรือ agent ก่อนหน้า และต้องทำงานร่วมกับสถานะนั้น

## รูปแบบการทำงานที่ต้องใช้

ก่อนแก้โค้ด ให้ตอบหรือบันทึกในงานให้ชัด:

- Requirement คืออะไร
- ขอบเขตงานรอบนี้คืออะไร
- ไฟล์ที่จะเปลี่ยนมีอะไรบ้าง
- แต่ละไฟล์มีหน้าที่อะไร
- ลำดับการทำงานคืออะไร
- ความเสี่ยงหรือข้อควรระวังคืออะไร

หลังแก้โค้ด ให้ตรวจ:

- `git diff`
- `npm run typecheck`
- `npm run lint`
- `npm run dev` เมื่อจำเป็นต้องตรวจ UI จริง

ถ้าคำสั่งใดรันไม่ได้ ต้องบอกสาเหตุให้ชัด และอย่าสรุปว่าผ่านแล้ว

## แนวทางเขียนโค้ด

- เขียนโค้ดให้อ่านง่าย เหมาะกับผู้เริ่มต้น
- ตั้งชื่อ function, component, variable ให้สื่อความหมาย
- แยก UI, business logic, data access และ auth ให้ชัดตามโครงสร้างเดิม
- ใช้ pattern ที่มีอยู่ใน repo ก่อนสร้าง abstraction ใหม่
- หลีกเลี่ยงโค้ดซับซ้อนเกินจำเป็น
- เพิ่ม comment เฉพาะจุดที่ logic อ่านยากจริง ๆ
- คำนึงถึง security และ maintainability ทุกครั้ง

## แนวทาง UI/UX ของ APS

- ระบบนี้ไม่ใช่แค่หน้าอัปโหลด transcript แต่เป็นระบบช่วยวางแผนการเรียนจนจบหลักสูตร
- หน้าแรกต้องตอบให้ได้ว่า APS ช่วยใคร เริ่มใช้อย่างไร และได้ผลลัพธ์อะไร
- Student Dashboard ควรจัดตามคำถามของนักศึกษา เช่น ตอนนี้เสี่ยงไหม เหลืออะไรบ้าง จะจบเมื่อไร ถ้าถอน/ไม่ผ่านจะกระทบไหม
- ปุ่ม "จัดการข้อมูลผลการเรียน" ต้องเห็นชัด เพราะเป็นจุดเริ่มต้นของการวิเคราะห์
- Admin Dashboard ควรเป็น workflow/setup checklist ไม่ใช่ CRUD ดิบ
- ใช้ภาษาไทยให้สม่ำเสมอและเป็นภาษาผู้ใช้ ไม่ใช่ศัพท์ technical เกินจำเป็น
- ลด UI ที่ดูเป็น template เช่น card ซ้ำ ๆ, gradient เยอะ, hierarchy ไม่ชัด

## ไฟล์และโฟลเดอร์ที่ควรระวัง

- `src/app/**`: Next.js routes และ pages
- `src/components/student/**`: UI ฝั่งนักศึกษา
- `src/components/admin/**`: UI ฝั่งผู้ดูแลระบบ
- `src/lib/analysis/**`: logic วิเคราะห์แผนการเรียน
- `src/lib/db/**`: data access
- `src/lib/auth/**`: auth/session/access control
- `prisma/schema.prisma`: database schema
- `docs/**`: เอกสารวิเคราะห์ แผนงาน คู่มือ และข้อมูลประกอบ final

โฟลเดอร์หรือไฟล์ชั่วคราว/generated ที่ไม่ควรถูกนำไปเป็นส่วนหลักของ app:

- `-d/**`
- `.next/**`
- `node_modules/**`
- `final_paragraphs*.txt`
- `final_body_esc.txt`
- `angle_paras*.txt`
- `thesis_deep.txt`
- `*.tsbuildinfo`
- `*.log`

## Checklist ก่อนส่งงานแต่ละรอบ

- Requirement ยังตรงกับสิ่งที่ผู้ใช้ขอ
- ไม่มีการแก้ไฟล์นอก scope โดยไม่จำเป็น
- ไม่มีการลบโค้ดเดิมโดยไม่มีเหตุผล
- ไม่มี package แปลก ๆ เพิ่มโดยไม่จำเป็น
- ไม่มี API key หรือข้อมูลลับหลุด
- `git diff` อ่านแล้วเข้าใจ
- `npm run typecheck` ผ่าน
- `npm run lint` ผ่านหรือมีคำอธิบาย warning ที่เหลือ
- ถ้าเป็นงาน UI ต้องเปิดเว็บจริงเพื่อตรวจ layout และ flow
