# อัปเดตระบบ APS สำหรับเอกสารประกอบการสอบ

วันที่อัปเดต: 03/06/2569

## 1. สิ่งที่แก้ไขในระบบล่าสุด

ระบบ Academic Planning Support (APS) ถูกปรับทิศทางจากเว็บที่เน้นอัปโหลดผลการเรียน ไปเป็นเว็บสนับสนุนการวางแผนการเรียนตามหลักสูตร โดยมีการปรับหน้าแรก หน้าแผงควบคุมนักศึกษา และหน้าผู้ดูแลระบบให้เป็นระบบสีสว่าง อ่านง่าย และใช้คำภาษาไทยมากขึ้น

สิ่งที่ปรับแล้ว:

- หน้าแรกและหน้าเข้าสู่ระบบ ปรับเป็นหน้าต้อนรับที่อธิบายวัตถุประสงค์ของระบบ และมีปุ่มเข้าสู่แดชบอร์ดวางแผนการเรียนชัดเจน
- หน้าแผงควบคุมนักศึกษา แยกส่วนข้อมูลเป็น สถานะนักศึกษา คาดการณ์วันจบ แผนรายเทอม วิชาที่ขวางการจบ ระบบจำลองสถานการณ์เรียน รายวิชาที่เปิดสอน และข้อมูลเสริม
- เครื่องมือจัดการไฟล์ผลการเรียนถูกแยกออกเป็นหน้าเฉพาะ `/student/transcript-tools` เพื่อลดความรกของหน้าแรก แต่จากการใช้งานจริงควรเพิ่มปุ่มลัดกลับไปยังส่วนอัปโหลด PDF ให้ชัดกว่านี้
- หน้าแผนผัง 8 ปี ปรับให้แสดงเป็นตารางรายปีและรายเทอม พร้อมสีสถานะของรายวิชา เช่น ผ่านแล้ว วางแผนไว้ ถอน/ไม่ผ่าน และวิชาที่ขวางการจบ
- หน้า What-if Simulation ปรับคำอธิบายเป็นภาษาไทย เช่น จำลองสถานการณ์เรียน วิชาที่เปิดให้ลงต่อได้ และวิชาที่ถูกขวางเพิ่ม
- หน้าผู้ดูแลระบบ ปรับจาก CRUD ดิบให้เป็น workflow ได้แก่ ตั้งค่าหลักสูตร จัดการรายวิชา จัดการวิชาบังคับก่อน แผนผังรายเทอม วิชาเปิดแต่ละเทอม และตรวจความพร้อม
- เพิ่มส่วน import CSV template สำหรับผู้ดูแลระบบ เพื่อให้สามารถนำเข้าข้อมูลหลักสูตร รายวิชา วิชาบังคับก่อน แผนรายเทอม วิชาเปิดสอน และกติกาได้โดยไม่ต้องแก้ JSON
- ปรับ theme รวมเป็น Light Mode ใช้พื้นหลัง slate-50, การ์ดสีขาว, sidebar สีกรมท่า และสีสถานะ sky/emerald/amber/red

ข้อผิดพลาด UX ที่พบจากการปรับล่าสุด:

- ปุ่มอัปโหลด PDF ไม่ควรถูกซ่อนจนผู้ใช้หาไม่เจอ แม้ระบบควรเน้นการวางแผนการเรียน แต่การนำเข้าข้อมูลผลการเรียนเป็นจุดเริ่มต้นสำคัญ จึงควรมีปุ่มลัดหรือแถบแจ้งเตือนบนหน้าแผงควบคุมเสมอ
- ควรเพิ่ม card ชื่อ “จัดการข้อมูลผลการเรียน” บนหน้า Student Dashboard โดยมีปุ่ม “อัปโหลด PDF / เพิ่มรายวิชาเอง” เพื่อให้ผู้ใช้เข้าถึงเครื่องมือนี้ได้ทันที

## 2. เนื้อหาที่ควรใส่ในบทที่ 2 ผลงานที่เกี่ยวข้อง

บทที่ 2 ควรเน้นการเปรียบเทียบระบบที่เกี่ยวข้องกับงาน APS โดยแบ่งเป็นกลุ่มดังนี้

### 2.1 ระบบสารสนเทศนักศึกษา

กล่าวถึงระบบทะเบียนหรือระบบสารสนเทศนักศึกษา ซึ่งทำหน้าที่เก็บข้อมูลรายวิชา ผลการเรียน ตารางเรียน และสถานะนักศึกษา ระบบเหล่านี้มักเน้นการแสดงข้อมูลย้อนหลังและข้อมูลปัจจุบัน แต่ไม่ได้ช่วยวิเคราะห์แผนการเรียนในอนาคตแบบรายบุคคล

ประเด็นที่เชื่อมโยงกับ APS:

- ใช้ข้อมูลผลการเรียนเป็นข้อมูลตั้งต้น
- ใช้ข้อมูลหลักสูตรและรายวิชาเป็นเงื่อนไขในการตรวจสอบ
- ยังมีข้อจำกัดด้านการช่วยวางแผนอนาคต เช่น ไม่บอกว่าถ้าถอนวิชาหรือตกวิชาตัวต่อจะจบช้ากี่เทอม

### 2.2 ระบบ Degree Audit

Degree Audit เป็นระบบตรวจสอบว่านักศึกษาผ่านรายวิชาและหน่วยกิตตามโครงสร้างหลักสูตรครบหรือไม่ จุดแข็งคือช่วยตรวจความครบถ้วนของหลักสูตร แต่หลายระบบยังไม่ได้จำลองแผนการเรียนรายเทอม หรือวิเคราะห์ผลกระทบจากวิชาบังคับก่อนอย่างละเอียด

ประเด็นที่ APS นำมาใช้:

- ตรวจรายวิชาที่ผ่านแล้วและยังเหลือ
- ตรวจหมวดวิชาและหน่วยกิตขั้นต่ำ
- ตรวจเงื่อนไขของวิชาบังคับก่อน

### 2.3 ระบบ Academic Planning และ What-if Simulation

ระบบวางแผนการเรียนบางระบบมีการจำลองสถานการณ์ เช่น เพิ่มรายวิชา ถอนรายวิชา หรือเปลี่ยนแผนการเรียน เพื่อดูผลกระทบต่อหน่วยกิตและกำหนดจบ APS นำแนวคิดนี้มาใช้ในบริบทของหลักสูตรภาควิชาคอมพิวเตอร์ โดยเน้นการตอบคำถามว่า “หากถอน/ลง/ไม่ผ่านรายวิชา จะกระทบวันจบอย่างไร”

สิ่งที่ APS แตกต่าง:

- ใช้ข้อมูลวิชาที่เปิดในแต่ละภาคการศึกษา
- ใช้เงื่อนไขวิชาบังคับก่อน
- แสดงผลเป็นแผนรายเทอมและแผน 8 ปี
- เน้นกลุ่มนักศึกษาที่มีความเสี่ยงติดโปรต่ำ

### 2.4 ระบบจัดการหลักสูตรสำหรับผู้ดูแลระบบ

ระบบที่ดีไม่ควรให้ผู้ดูแลระบบแก้ข้อมูลด้วย JSON หรือโค้ดโดยตรง แต่ควรมีหน้าจอสำหรับจัดการหลักสูตร รายวิชา วิชาบังคับก่อน แผนการเรียน และวิชาเปิดแต่ละเทอม APS จึงออกแบบ admin workflow เพื่อให้แก้ข้อมูลได้ผ่าน UI และรองรับการนำเข้า CSV/Excel template

### 2.5 สรุปช่องว่างของระบบเดิม

จากการศึกษาระบบที่เกี่ยวข้อง พบช่องว่างสำคัญ ได้แก่

- ระบบทั่วไปแสดงข้อมูลผลการเรียน แต่ไม่ช่วยวางแผนจนจบแบบรายเทอม
- ระบบทั่วไปไม่เน้น What-if Simulation สำหรับถอน/เพิ่ม/ไม่ผ่านรายวิชา
- ระบบทั่วไปไม่แสดงผลกระทบจากวิชาบังคับก่อนแบบอ่านง่าย
- ผู้ดูแลระบบมักต้องจัดการข้อมูลหลังบ้านด้วยวิธีที่ไม่เป็นมิตร
- ระบบที่เน้น OCR อาจไม่แม่นยำกับเอกสารทุกแบบ จึงควรมีขั้นตอน preview และ manual correction เพื่อให้การนำเข้าข้อมูลสำเร็จได้

## 3. เนื้อหาที่ควรใส่ในบทที่ 3 ทฤษฎีและความรู้ที่เกี่ยวข้อง

### 3.1 ระบบสารสนเทศเพื่อสนับสนุนการตัดสินใจ

APS เป็นระบบสนับสนุนการตัดสินใจสำหรับนักศึกษาและผู้ดูแลระบบ โดยนำข้อมูลผลการเรียน หลักสูตร รายวิชา และเงื่อนไขการเปิดสอนมาวิเคราะห์ เพื่อช่วยให้นักศึกษาตัดสินใจวางแผนการเรียนได้ดีขึ้น

### 3.2 การวิเคราะห์หลักสูตร

การวิเคราะห์หลักสูตรประกอบด้วยการตรวจสอบ:

- รายวิชาที่ผ่านแล้ว
- รายวิชาที่ยังไม่ผ่านหรือยังไม่ได้เรียน
- หมวดวิชาและหน่วยกิตขั้นต่ำ
- รายวิชาบังคับก่อน
- วิชาที่เปิดในแต่ละภาคการศึกษา
- แผนการเรียนปกติของแต่ละหลักสูตร

### 3.3 วิชาบังคับก่อนและความสัมพันธ์ของรายวิชา

วิชาบังคับก่อนเป็นเงื่อนไขที่กำหนดว่านักศึกษาต้องผ่านรายวิชาหนึ่งก่อนจึงจะสามารถลงอีกวิชาหนึ่งได้ ในระบบ APS ความสัมพันธ์นี้ถูกใช้ในการวิเคราะห์วิชาที่ขวางการจบ และใช้ในการจำลองสถานการณ์หากนักศึกษาถอนหรือตกวิชา

### 3.4 Graduation Forecast

Graduation Forecast คือกระบวนการคาดการณ์ว่านักศึกษาจะจบการศึกษาได้ในปีการศึกษาและภาคการศึกษาใด โดยพิจารณาจาก:

- รายวิชาที่ผ่านแล้ว
- รายวิชาที่ยังเหลือ
- จำนวนหน่วยกิตสูงสุดต่อภาคการศึกษา
- วิชาที่เปิดในแต่ละเทอม
- เงื่อนไขวิชาบังคับก่อน
- เทอมปัจจุบันของนักศึกษา

### 3.5 What-if Simulation

What-if Simulation คือการจำลองสถานการณ์ก่อนตัดสินใจจริง เช่น

- ถอนรายวิชาแล้วจะจบช้ากี่เทอม
- ลงรายวิชาเพิ่มแล้วเปิดให้ลงวิชาต่อใดได้บ้าง
- หากไม่ผ่านวิชาบังคับก่อน จะทำให้วิชาใดถูกขวาง
- เปรียบเทียบแผนเดิมกับแผนหลังจำลอง

### 3.6 สถานะติดโปรและความเสี่ยงทางการเรียน

ระบบ APS แสดงสถานะนักศึกษา เช่น ปกติ โปรสูง โปรต่ำ และเสี่ยงโปรต่ำในเทอมถัดไป โดยอธิบายเหตุผลประกอบ เช่น GPAX ต่ำกว่าเกณฑ์ หน่วยกิตผ่านน้อย ถอนหรือตกวิชาตัวต่อ

### 3.7 UX/UI สำหรับระบบสนับสนุนการเรียน

ระบบสำหรับนักศึกษาควรมี UI ที่เข้าใจง่าย ไม่ซับซ้อน และไม่ทำให้ผู้ใช้รู้สึกว่ากำลังใช้งานระบบหลังบ้าน ข้อมูลสำคัญควรอยู่ในหน้าแรก ได้แก่ สถานะนักศึกษา คาดการณ์วันจบ แผนรายเทอม วิชาที่ขวางการจบ และปุ่มจัดการข้อมูลผลการเรียน

## 4. เนื้อหาที่ควรใส่ในบทที่ 5 การวิเคราะห์และออกแบบงานระบบ

### 5.1 ภาพรวมระบบ

ระบบ APS แบ่งผู้ใช้ออกเป็น 2 กลุ่ม ได้แก่ นักศึกษาและผู้ดูแลระบบ นักศึกษาใช้ระบบเพื่อดูสถานะ วางแผนการเรียน คาดการณ์วันจบ และจำลองสถานการณ์เรียน ส่วนผู้ดูแลระบบใช้จัดการข้อมูลหลักสูตร รายวิชา วิชาบังคับก่อน แผนการเรียน และวิชาเปิดแต่ละเทอม

### 5.2 กระบวนการทำงานฝั่งนักศึกษา

1. นักศึกษาเข้าสู่ระบบด้วยบัญชีมหาวิทยาลัย
2. เลือกหรือยืนยันหลักสูตรและกลุ่มแผนการเรียน
3. อัปโหลด PDF ผลการเรียน หรือเพิ่มรายวิชาด้วยตนเอง
4. ระบบอ่านข้อมูลเท่าที่อ่านได้ และแสดงหน้าตรวจสอบก่อนบันทึก
5. นักศึกษาตรวจและแก้ไขแถวที่ผิด เช่น รหัสวิชา ชื่อวิชา เทอม ปี และเกรด
6. นักศึกษายืนยันข้อมูลผลการเรียน
7. ระบบวิเคราะห์สถานะนักศึกษา แผนจบ วิชาที่ขวาง และแผนรายเทอม
8. นักศึกษาสามารถจำลองสถานการณ์เรียนเพื่อเปรียบเทียบแผนเดิมกับแผนจำลอง

### 5.3 กระบวนการทำงานฝั่งผู้ดูแลระบบ

1. ผู้ดูแลระบบเข้าสู่ระบบ
2. ตั้งค่าหลักสูตรและโครงสร้างหน่วยกิต
3. จัดการรายวิชา
4. จัดการวิชาบังคับก่อนและวิชาตัวต่อ
5. จัดการแผนการเรียนรายปี/รายเทอม
6. จัดการรายวิชาที่เปิดสอนในแต่ละเทอม
7. นำเข้าข้อมูลด้วย CSV template ได้
8. ตรวจความพร้อมของข้อมูลก่อนเผยแพร่ให้ฝั่งนักศึกษาใช้งาน

### 5.4 โครงสร้างฐานข้อมูล

ฐานข้อมูลออกแบบให้รองรับหลักสูตรอื่นในอนาคต ไม่จำกัดเฉพาะ CS2565 และ IT2565 โดยมีตารางสำคัญ ได้แก่

- User: ข้อมูลผู้ใช้และบทบาท
- Program: ข้อมูลหลักสูตร
- Course: รายวิชา
- ProgramStructure: โครงสร้างหมวดวิชาและหน่วยกิตขั้นต่ำ
- Prerequisite: ความสัมพันธ์วิชาบังคับก่อน
- StudyPlan: แผนรายปี/รายเทอม
- CourseOffering: วิชาที่เปิดสอนในแต่ละปีและเทอม
- TranscriptUpload: ประวัติการนำเข้าข้อมูลผลการเรียน
- StudentGrade: รายวิชาที่นักศึกษาเรียนและผลการเรียน
- TranscriptSemesterSummary: สรุป GPA/GPAX รายเทอม
- AnalysisResult และ Recommendation: ผลการวิเคราะห์และคำแนะนำ
- SimulationSession และ SimulationCourse: การจำลองผลการเรียน

### 5.5 การออกแบบหน้าจอ

หน้าจอนักศึกษาควรจัดลำดับจากข้อมูลสำคัญไปหาข้อมูลแก้ไข:

1. สถานะนักศึกษา
2. คาดการณ์วันจบ
3. แผนรายเทอม
4. วิชาที่ขวางการจบ
5. ระบบจำลองสถานการณ์เรียน
6. ปุ่มลัดจัดการข้อมูลผลการเรียน

หน้าจอผู้ดูแลระบบควรเป็น workflow ไม่ใช่ตาราง CRUD ดิบ เพื่อให้ผู้ดูแลระบบเข้าใจว่าต้องกรอกข้อมูลอะไรและข้อมูลใดจำเป็นต่อการวิเคราะห์

## 5. เนื้อหาที่ควรใส่ในบทที่ 6 ผลการดำเนินงาน ข้อจำกัด และข้อเสนอแนะ

### 6.1 ผลการดำเนินงาน

ระบบ APS สามารถพัฒนาฟังก์ชันหลักได้ดังนี้

- นักศึกษาสามารถเข้าสู่ระบบและดูแผงควบคุมการวางแผนการเรียน
- ระบบสามารถแสดงสถานะนักศึกษา เช่น ปกติ โปรสูง โปรต่ำ และความเสี่ยง
- ระบบสามารถคาดการณ์แผนการเรียนจนจบโดยใช้ข้อมูลหลักสูตร รายวิชาที่ผ่านแล้ว รายวิชาที่ยังเหลือ วิชาบังคับก่อน และวิชาที่เปิดในแต่ละเทอม
- ระบบสามารถแสดงแผนรายเทอมและแผนผัง 8 ปี
- ระบบสามารถแสดงวิชาที่ขวางการจบจากเงื่อนไขวิชาบังคับก่อน
- ระบบสามารถจำลองสถานการณ์เรียน เช่น ถอนวิชา ลงวิชาเพิ่ม หรือไม่ผ่านวิชา แล้วเปรียบเทียบแผนเดิมกับแผนจำลอง
- ผู้ดูแลระบบสามารถจัดการหลักสูตร รายวิชา วิชาบังคับก่อน แผนการเรียน และวิชาเปิดสอนในแต่ละเทอมผ่าน UI
- ระบบรองรับการนำเข้าข้อมูลหลักสูตรด้วย CSV template
- ระบบมีขั้นตอนตรวจและแก้ไขข้อมูลผลการเรียนก่อนบันทึก เพื่อให้การนำเข้าข้อมูลไม่ขึ้นอยู่กับการอ่าน PDF อย่างเดียว

### 6.2 ข้อจำกัด

- การอ่าน PDF ผลการเรียนยังไม่สามารถรับประกันได้ 100% กับทุกไฟล์ โดยเฉพาะไฟล์ที่เป็นภาพสแกนหรือมีรูปแบบ text extraction ที่ไม่รองรับ
- จากการทดสอบ มีไฟล์บางฉบับที่อ่านไม่ได้หรืออ่านได้ไม่ครบ จึงต้องใช้ขั้นตอนเพิ่ม/แก้ไขรายวิชาด้วยตนเอง
- UI ยังต้องปรับปรุง โดยเฉพาะปุ่มอัปโหลด PDF ที่ไม่ควรถูกซ่อนจากหน้าแรกของนักศึกษา
- ข้อมูลหลักสูตรและแผนรายเทอมยังเป็นข้อมูลตัวอย่างจาก CS2565/IT2565 จึงต้องตรวจความครบถ้วนกับเอกสารหลักสูตรจริงก่อนใช้งานจริง
- เกณฑ์สถานะติดโปรและเกณฑ์ทางวิชาการต้องตรวจสอบกับระเบียบมหาวิทยาลัยให้ถูกต้อง
- ระบบยังไม่ได้ครอบคลุมทุกกรณีเฉพาะ เช่น การเทียบโอน รายวิชาแทน รายวิชาที่นับซ้ำไม่ได้ หรือหลักสูตรที่เปลี่ยนแปลงระหว่างปี

### 6.3 ข้อเสนอแนะ

- เพิ่มปุ่ม “อัปโหลด/แก้ไขข้อมูลผลการเรียน” กลับไว้บนหน้า Student Dashboard ในตำแหน่งที่เห็นชัด
- ปรับ UI ด้วยการทดสอบกับผู้ใช้จริง เพื่อดูว่านักศึกษาหาเมนูสำคัญเจอหรือไม่
- เพิ่มการนำเข้า Excel สำหรับทั้งฝั่งนักศึกษาและผู้ดูแลระบบ
- เพิ่มตัวช่วยแปลง PDF หลักสูตรเป็นข้อมูลตั้งต้น แต่ยังคงให้ผู้ดูแลระบบตรวจและยืนยันก่อนเผยแพร่
- เพิ่มระบบตรวจคุณภาพข้อมูลหลังบ้าน เช่น รายวิชาไม่มีเทอมเปิด วิชาบังคับก่อนหาย แผนรายเทอมไม่ครบ หรือหน่วยกิตรวมไม่ถึง
- เพิ่มรายงานสำหรับอาจารย์ที่ปรึกษา เพื่อดูนักศึกษากลุ่มเสี่ยงโปรต่ำและวิชาที่มักขวางการจบ
- เพิ่มชุดทดสอบจากไฟล์จริง เช่น CS_F, CS_Neoi, CS_Nook, IT_Sand และ CS_Minie เพื่อใช้ยืนยันผลก่อนสอบจริง

## 6. ข้อความสรุปสำหรับใส่รายงาน

ระบบที่พัฒนาขึ้นมีเป้าหมายเพื่อสนับสนุนการวางแผนการเรียนตามหลักสูตร โดยใช้ข้อมูลผลการเรียนของนักศึกษา ข้อมูลหลักสูตร รายวิชา วิชาบังคับก่อน และข้อมูลรายวิชาที่เปิดสอนในแต่ละภาคการศึกษา เพื่อคาดการณ์ภาคการศึกษาที่นักศึกษาจะสำเร็จการศึกษา แสดงรายวิชาที่ต้องเรียนในแต่ละเทอม วิเคราะห์รายวิชาที่ขวางการจบ และจำลองสถานการณ์เรียนก่อนตัดสินใจลงทะเบียน ระบบยังออกแบบให้ผู้ดูแลระบบสามารถจัดการข้อมูลหลักสูตรและข้อมูลประกอบการวิเคราะห์ผ่านหน้าจอหรือ CSV template ได้โดยไม่จำเป็นต้องแก้ไขโค้ดโดยตรง
